# cli-consultants v2.0 Design — Triple-Mandatory Final-Pass + Scorecard Schema

**Status:** Approved (brainstorm 2026-05-13)
**Branch:** `v2.0` (worktree `.worktrees/v2.0`)
**Bumps:** 1.2.0 → 2.0.0 (breaking — mandatory policy change + scorecard schema break)

## Goal

Promote Plan-subagent from "optional bonus voice" to **mandatory third channel** on every final-pass when all three consultants are configured (Full mode). Extend scorecard schema to three columns. Strictly policy/documentation change — no new CLI, no new wrapper scripts, no runtime detection changes beyond v1.2.0.

## Out of scope (parked for v2.1)

- **Spec categorization** → Gemini mandatory on iter-1..N-1 for UX/codegen/operational categories
- **`.agent/SCORECARD_INSIGHTS.md`** companion file (compressed scorecard patterns)
- **Categorization-driven dynamic mandatory rules** (per-spec policy)

These remain in the May 2026 update document; v2.0 ships only the structural triple change and scorecard schema bump.

## Architecture

Two coordinated changes, both pure-text policy:

### A. Triple-mandatory final-pass (Full mode)

Before (v1.2.0):
- Spec/plan final-pass: Codex **mandatory** + Gemini **mandatory (parallel)** + Plan-subagent **optional bonus voice**
- Tie-break: Plan-subagent **mandatory deciding voice**
- Codex rate-limited / Gemini drifting: Plan-subagent **mandatory replacement**

After (v2.0):
- Spec/plan/impl-verify final-pass: Codex **mandatory** + Gemini **mandatory (parallel)** + Plan-subagent **mandatory (parallel)**
- Tie-break, replacement scenarios: unchanged (already mandatory in v1.2.0)

In **Codex-only-plus-Plan mode** (Gemini wrapper absent): unchanged from v1.2.0 — Plan-subagent already compensates as second voice. Triple-mandatory only meaningfully applies in Full mode.

In **Plan-subagent unavailable** edge case (environment without Agent tool — rare; effectively means not in Claude Code session): the operational skill is moot in that environment. Surface to user and fall back to v1.2.0 dual final-pass with explicit `Plan-subagent UNAVAILABLE (no Agent tool in this runtime)` disclosure. New disclosure flag added.

### B. Scorecard schema bump

Template `consultant_scorecard.md` header updated:

- New columns:
  - `plan_subagent_tldr` — 1–2 line summary of Plan-subagent's answer
  - `plan_subagent_eval` — `correct / partial / wrong / irrelevant / timeout`
- Extended `winner` value set: `codex / gemini / plan_subagent / tie / all-wrong`
- Trigger description updated: "at least 2 of 3 consultants engaged on the same question" (formerly "both consultants"). In Full mode v2.0 this fires on every spec/plan/impl-verify final-pass (since all three are mandatory); in Codex-only-plus-Plan mode it fires on every final-pass (Codex + Plan-subagent = 2 of 3).

**Backwards compat (per brainstorm Q2):** Existing scorecard files in teammate projects remain untouched. Setup skill on re-run:
- If `.agent/consultant_scorecard.md` exists with v1.x header → emit one-line notice "Existing scorecard uses v1.x schema; v2.0 entries will append below with the new columns" and leave the file alone.
- New installations get v2.0-schema scorecard from the template.

No data migration script. No file replacement. The agent tolerates mixed-version files in practice (it can read both columnsets).

## Data flow — final-pass orchestration

Unchanged from v1.2.0 mechanically — just elevated to mandatory:

```bash
# Full mode triple final-pass
[ -x tools/ask_codex.sh ]  && [ -x tools/ask_gemini.sh ] && PLAN_SUBAGENT_AVAILABLE=1
# In the SAME orchestration turn:
./tools/ask_codex.sh  > /tmp/codex_run.log  2>&1 &     # bash background
./tools/ask_gemini.sh > /tmp/gemini_run.log 2>&1 &     # bash background
# Plan-subagent: dispatched as Agent tool call in the same message that fires
# the wrappers — agent runtime parallelizes natively.
wait  # for bash backgrounds
# Synthesize all three; report triple convergence/divergence + scorecard row.
```

The v1.2.0 orchestration block in `using-cli-consultants/SKILL.md` is already the right pattern; v2.0 changes the wording from "when running Plan-subagent alongside as bonus" to "in Full mode, the canonical pattern is this triple — there is no dual-without-Plan-subagent variant in Full mode".

## Error handling

- **Codex 429 in Full mode:** Per v1.2.0 — Plan-subagent already mandatory replacement; report mentions "Codex unreachable, Plan-subagent took primary slot, Gemini ran in parallel as second voice".
- **Gemini drift in Full mode:** Per v1.2.0 — Plan-subagent replaces; report shows "Codex ran; Gemini drifted (discarded); Plan-subagent took second slot".
- **Plan-subagent fails (rare):** v2.0 needs a documented behavior. Default: continue with Codex + Gemini dual, note "Plan-subagent unreachable — fell back to v1.2.0 dual policy for this pass". Add this to failure-mode table.
- **Two consultants down simultaneously:** Continue with remaining one + explicit escalation to user — discipline is preserved through disclosure, not blocked execution.

## Testing strategy

Three verify scenarios (RED→GREEN), dispatched as `general-purpose` subagents reading the updated `skills/using-cli-consultants/SKILL.md` directly:

- **V1 — Triple-mandatory in Full mode:** Subagent told it's doing a spec final-pass with Codex+Gemini+Plan-subagent all configured. Must dispatch all three in parallel. Report must explicitly state "triple final-pass" and synthesize three answers, not two.
- **V2 — "Plan-subagent is optional bonus" rationalization:** Subagent under deadline pressure told "Plan-subagent is optional, can we skip to save tokens?". Must refuse — Plan-subagent is now mandatory in Full mode final-pass. Cite the new rationalization counter.
- **V3 — Scorecard trigger after triple-mandatory:** Subagent told a triple final-pass just completed (Codex + Gemini + Plan-subagent each ran, mild divergence on one finding). Must append a row to scorecard using v2.0 schema with `plan_subagent_tldr`, `plan_subagent_eval`, and a `winner` value from the extended set.

Each scenario validates one of the two structural changes (A or B). All three must PASS before merging the worktree branch to main.

## Risks

- **Cost increase.** Every final-pass now spawns a Plan-subagent call in addition to two bash wrappers. Plan-subagent costs current-session tokens (no separate CLI quota). For a 1100-line plan, the priming cost can be 5–10k tokens per call. Mitigation: this is the explicit trade for catching execution-readiness bugs.
- **Compliance via discipline alone.** No runtime enforcement — agent could lie about whether Plan-subagent ran. Mitigation: same rationalization-counter + red-flag mechanism that v1.x relies on. We don't add hooks (rejected as Approach #3 in brainstorm — out of scope).
- **Schema break for teams accumulating scorecards.** Mitigated by Q2 decision: don't touch existing files; new entries use new schema; mixed history is acceptable.
- **Breaking change unannounced.** Teammates updating via `/plugin marketplace update` get v2.0 silently. Mitigation: README + plugin description make the change loud. Optional: include a `MIGRATION.md` (decision deferred to plan stage).
